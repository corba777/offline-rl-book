Глава 3: Консервативное Q-обучение (CQL)

«Не доверяй значениям, которых не видел. А если не видел — занижай их.»

Проблема, переформулированная

В главе 2 мы увидели, что стандартное Q-обучение на офлайн данных порождает катастрофически оптимистичные Q-значения для OOD-действий. Жадная политика эксплуатирует эти завышенные значения — и терпит неудачу при развёртывании.

Conservative Q-Learning (CQL) — Kumar et al., NeurIPS 2020 — исправляет это одной идеей: добавить штраф, явно занижающий Q-значения для OOD-действий и одновременно сохраняющий их для действий из датасета.

Результат: Q-функция, пессимистичная для OOD-действий по построению. Жадная политика, видя низкие значения вне датасета, естественно остаётся в его пределах.

Идея

Стандартное TD-обучение минимизирует:

$$\mathcal{L}_{TD}(\theta) = \mathbb{E}_{(s,a,r,s') \sim \mathcal{D}} \left[ \left( r + \gamma \max_{a'} Q_{\bar\theta}(s', a') - Q_\theta(s, a) \right)^2 \right]$$

CQL добавляет консервативный штраф:

$$\mathcal{L}_{CQL}(\theta) = \mathcal{L}_{TD}(\theta) + \alpha \cdot \underbrace{\left( \mathbb{E}_{s \sim \mathcal{D},\, a \sim \mu} \left[ Q_\theta(s, a) \right] - \mathbb{E}_{(s,a) \sim \mathcal{D}} \left[ Q_\theta(s, a) \right] \right)}_{\text{OOD вниз, датасет вверх}}$$

где $\mu$ — предложенное распределение (равномерное или текущая политика), $\alpha > 0$ — сила консерватизма.

Коротко: минимизировать Q везде, но максимизировать в точках датасета.

Формализация

Полная целевая функция

CQL минимизирует регуляризованную ошибку Беллмана:

$$\min_\theta \, \alpha \left( \mathbb{E}_{s \sim \mathcal{D}} \left[ \log \sum_a \exp Q_\theta(s, a) \right] - \mathbb{E}_{(s,a) \sim \mathcal{D}} \left[ Q_\theta(s, a) \right] \right) + \mathcal{L}_{TD}(\theta)$$

Замечание: в оригинальной статье написано $\frac{1}{2}\mathcal{L}_{TD}$ — соглашение о нормировке, поглощаемое $\alpha$ и скоростью обучения. Наша реализация его опускает.

Член log-sum-exp — гладкая аппроксимация $\max_a Q(s,a)$, опускающая всю Q-поверхность вниз. Второй член поднимает Q именно в точках датасета.

Для непрерывных пространств действий logsumexp аппроксимируется через importance sampling с двумя предложениями — случайными равномерными действиями и действиями текущей политики:

$$\log \sum_a \exp Q(s, a) \approx \log \mathbb{E}_{a \sim \mu} \left[ \frac{\exp Q(s,a)}{\mu(a|s)} \right]$$

Зачем вычитать $\log \mu(a|s)$ в коде? При $\mu = \pi_\theta$ importance-sampling оценка раскрывается как $\text{logsumexp}_i[Q(s,a_i) - \log\pi_\theta(a_i|s)] + \text{const}$. Вычитание лог-вероятности предложения реализует коррекцию importance weights — без неё мы бы считали простое среднее Монте-Карло по $Q$, а не мягкий максимум, который требует CQL. Для случайных равномерных действий коррекция не нужна: их лог-плотность константна и сокращается в logsumexp.

Именно это делает compute_cql_loss() в cql.py.

Теоретическая гарантия

Теорема (Kumar et al., 2020). При подходящем $\alpha$ Q-функция, обученная CQL, удовлетворяет:

$$\hat{Q}^\pi(s, a) \leq Q^\pi(s, a) \quad \forall (s, a) \in \mathcal{D}$$

CQL является нижней оценкой истинной Q-функции в точках датасета. Политика, обученная на этой пессимистичной оценке, не может эксплуатировать завышенные значения — их просто нет.

Среда: ThermalProcessEnv

Игрушечная среда имитирует процесс нанесения асфальтного покрытия. Состояние — [температура, наполнитель, вязкость], нормализованное в [0, 1]. Действия — [delta_нагрева, delta_потока] в [-1, 1]. Динамика первого порядка с аддитивным шумом:

class ThermalProcessEnv:
    def __init__(self, T_target=0.6, f_target=0.5, max_steps=200,
                 noise_std=0.02, seed=42):
        self.alpha_T, self.beta_T = 0.92, 0.08
        self.alpha_f, self.beta_f = 0.88, 0.12

    def _viscosity(self, T, f):
        """Нелинейно: высокая температура -> низкая вязкость."""
        return float(np.clip(0.8 - 0.5 * T + 0.4 * f + 0.1 * f * (1 - T), 0, 1))

    def step(self, action):
        heat_in = action[0] * 0.5 + 0.5   # масштаб [-1,1] -> [0,1]
        T_new = np.clip(self.alpha_T * T + self.beta_T * heat_in
                        + self.rng.normal(0, self.noise_std), 0, 1)
        reward = (-2.0 * (T_new - self.T_target)**2
                  - 2.0 * (f_new - self.f_target)**2
                  - 0.1 * float(np.sum(action**2)))

Датасет: шумный PID

Данные собираются пропорциональным регулятором с гауссовым шумом (noise_scale=0.3) — смешанного качества, как реальные промышленные логи:

def pid_action(state, T_target, f_target):
    T, f, _ = state
    return np.array([
        np.clip(2.0 * (T_target - T), -1, 1),
        np.clip(2.0 * (f_target - f), -1, 1),
    ], dtype=np.float32)

def collect_offline_dataset(n_episodes=400, noise_scale=0.3, seed=0):

Сети

QNetwork принимает конкатенацию (state, action) и возвращает скаляр. GaussianPolicy использует stochastic actor в стиле SAC с tanh-сжатием — выходы гарантированно в [-1, 1]:

class QNetwork(nn.Module):
    def forward(self, state, action):
        return self.net(torch.cat([state, action], -1)).squeeze(-1)


class GaussianPolicy(nn.Module):
    def sample(self, state):
        """Возвращает (action в [-1,1], log_prob) через репараметризацию."""
        mean, std = self._dist(state)
        raw    = mean + std * torch.randn_like(mean)
        action = torch.tanh(raw)
        lp     = (torch.distributions.Normal(mean, std).log_prob(raw)
                  - torch.log(1 - action.pow(2) + 1e-6)).sum(-1)
        return action, lp

Член log(1 - tanh²(x) + ε) корректирует логарифм вероятности на якобиан преобразования tanh.

Функция потерь CQL

Ядро файла. Принимает два Q-networks и их таргеты (double-Q), политику и батч переходов:

def compute_cql_loss(Q, Q_other, Q_tgt, Q_other_tgt, policy,
                     states, actions, rewards, next_states, dones,
                     alpha_cql=1.0, alpha_ent=0.1, gamma=0.99, n_samples=10):

    # ── TD цель (SAC обновление Беллмана) ────────────────────────────────
    with torch.no_grad():
        a_next, lp_next = policy.sample(next_states)
        v_next = (torch.min(Q_tgt(next_states, a_next),
                            Q_other_tgt(next_states, a_next))
                  - alpha_ent * lp_next)
        td_target = rewards + gamma * (1 - dones) * v_next

    q_data  = Q(states, actions)
    td_loss = F.mse_loss(q_data, td_target)

    # ── CQL штраф ────────────────────────────────────────────────────────
    # Расширяем состояния для векторизованного сэмплирования
    s_rep = states.unsqueeze(1).expand(-1, n_samples, -1).reshape(B * n_samples, -1)

    # Случайные OOD-действия
    a_rand = torch.FloatTensor(B * n_samples, actions.shape[-1]).uniform_(-1, 1)

    # Действия политики (с importance weighting)
    a_pi, lp_pi = policy.sample(s_rep)

    q_rand = Q(s_rep, a_rand).reshape(B, n_samples)
    q_pi   = (Q(s_rep, a_pi) - lp_pi.detach()).reshape(B, n_samples)

    # logsumexp по OOD-сэмплам — член «опустить вниз»
    logsumexp   = torch.logsumexp(torch.cat([q_rand, q_pi], dim=1), dim=1)
    cql_penalty = (logsumexp - q_data).mean()

    return td_loss + alpha_cql * cql_penalty, {...}

На что обратить внимание:

Double-Q в TD-цели. torch.min(Q_tgt, Q_other_tgt) снижает переоценку в обновлении независимо от CQL.

Два OOD-предложения. Случайные равномерные действия покрывают всё пространство. Действия политики добавляют importance weighting — q_pi - lp_pi аппроксимирует $\log \mathbb{E}_\mu[\exp Q / \mu]$.

q_data — якорь. TD-потеря удерживает Q(s,a) близко к цели Беллмана в точках датасета. CQL-штраф гарантирует, что logsumexp OOD-значений остаётся ниже.

CQLAgent

Агент объединяет double-Q, политику и опциональную автоматическую настройку $\alpha$ через двойственный градиентный спуск:

class CQLAgent:
    def update(self, batch):
        # Обновляем оба Q-networks
        l1, i1 = compute_cql_loss(self.Q1, self.Q2, self.Q1_tgt, self.Q2_tgt, ...)
        l2, i2 = compute_cql_loss(self.Q2, self.Q1, self.Q2_tgt, self.Q1_tgt, ...)
        (l1 + l2).backward()

        # Auto alpha: поддерживаем штраф около target_cql
        if self.auto_alpha:
            pen = (i1['cql_penalty'] + i2['cql_penalty']) / 2
            al  = -self.log_alpha * (pen - self.target_cql)
            al.backward()

        # Политика: максимизируем Q - энтропия
        a_pi, lp = self.policy.sample(s)
        q_pi = torch.min(self.Q1(s, a_pi), self.Q2(s, a_pi))
        lpi  = (self.alpha_ent * lp - q_pi).mean()

        # Мягкое обновление таргетов: θ_target ← (1-τ)θ_target + τθ
        for p, pt in zip(self.Q1.parameters(), self.Q1_tgt.parameters()):
            pt.data.mul_(1 - self.tau).add_(self.tau * p.data)

Гиперпараметр alpha

$\alpha$ Поведение
$\alpha = 0$ Стандартный SAC — нет консерватизма
$\alpha = 1.0$ Хорошая точка старта
$\alpha = 5$–$10$ Сильный консерватизм — политика близка к $\pi_\beta$
$\alpha \to \infty$ Эквивалентно поведенческому клонированию

Автоматическая настройка через Лагранжиан: удерживаем штраф CQL вблизи цели $\tau$. В CQLAgent задайте auto_alpha=True и target_cql=-1.0.

Замечание об энтропийном коэффициенте. В реализации используется фиксированное alpha_ent=0.1. В полноценном CQL этот коэффициент тоже настраивается автоматически — с целью поддерживать $\mathcal{H}^* = -\dim(\mathcal{A})$ (одна нат на измерение действия), как принято в SAC. Фиксированное значение подходит для задачи управления тепловым процессом; для других сред рекомендуется автонастройка.

Интуиция: ландшафт Q-функции

Представьте Q-функцию как ландшафт над пространством действий. Стандартное Q-обучение оставляет этот ландшафт неограниченным между точками датасета — он стремится расти из-за оптимистичного обобщения.

CQL добавляет гравитацию: тянет весь ландшафт вниз, пока TD-потеря удерживает его высоким в точках датасета. Жадная политика видит высокие значения только у действий из датасета и естественно остаётся in-distribution.

Как выбрать α

$\alpha$ — важнейший гиперпараметр CQL. Правильное значение зависит от качества датасета, масштаба наград и доверия к поведенческой политике. Вот структурированный подход.

Шаг 1 — Проверьте знак CQL-штрафа

Перед настройкой $\alpha$ убедитесь, что штраф корректно положителен: $\mathbb{E}_\mu[Q] > \mathbb{E}_D[Q]$ должно выполняться после нескольких сотен обновлений. Если штраф отрицателен с самого начала — сэмплирование OOD-действий сломано.

Шаг 2 — Грид-поиск по логарифмической шкале

for alpha in [0.1, 0.5, 1.0, 5.0, 10.0]:
    agent = CQLAgent(..., alpha_cql=alpha)
    train_agent(agent, loader, n_epochs=80)
    res = evaluate(agent, env, ...)
    print(f"alpha={alpha:.1f}  reward={res['reward_mean']:.2f}")

Пять прогонов дают грубую карту компромисса награда–консерватизм.

Когда использовать автонастройку

Используйте auto_alpha=True если: (а) политика будет развёрнута в реальной системе и безопасность важнее производительности, или (б) датасет охватывает несколько режимов работы с разными масштабами наград. Задайте target_cql небольшим отрицательным значением (например, -1.0).

Используйте фиксированный $\alpha$ если: проводите офлайн-эксперименты с воспроизводимыми результатами, или уже нашли хорошее значение грид-поиском.

Практические советы

Нормализуйте наблюдения по статистикам датасета. CQL чувствителен к масштабу.

Мониторьте CQL-штраф (E_μ[Q] - E_D[Q]). Он должен быть положительным и стабильным. Если уходит в минус — увеличивайте $\alpha$.

Всегда используйте double-Q. torch.min(Q1, Q2) снижает переоценку независимо от CQL — оба механизма дополняют друг друга.

Для данных с несколькими режимами работы добавьте режим как признак состояния или обучите отдельные агенты на каждый режим.

Ограничения

Требует разметки наград. В отличие от BC, CQL нужны кортежи $(s, a, r, s')$. Если датасет содержит только $(s, a, s')$ — необходимо определить функцию вознаграждения.

Чувствителен к масштабу наград. Баланс между TD-потерей и CQL-штрафом зависит от величины наград. Если награды большие (например, в тысячах), TD-потеря доминирует и CQL теряет эффект. Нормализуйте награды в $[-1, 1]$ или $[0, 1]$.

Консервативен по построению. CQL не превзойдёт поведенческую политику значительно в регионах с редкими данными. Он создан для безопасности, а не для агрессивной экстраполяции. Для задач, требующих существенного выхода за пределы датасета, более уместны модельные методы (Глава 5).

Непрерывные пространства действий требуют сэмплирования. Аппроксимация logsumexp требует сэмплирования — как правило, 10 случайных + 10 политиковых действий на состояние. Это увеличивает вычислительные затраты по сравнению с BC или TD3+BC.
